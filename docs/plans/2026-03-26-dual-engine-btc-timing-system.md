# Helen — 极简双引擎量化系统 v2.0 — BTC 择时仓位管理

> **设计日期**: 2026-03-26
> **版本**: v2.0（v1.1 + 三刀优化：砍风控/贪婪映射/自然交接棒）
> **标的**: BTC (Binance BTCUSDT)，可复用于单只美股/指数
> **频率**: 4H K 线轮询（BTC 24/7，每天 6 根）
> **数据源**: Binance API (免费)
> **与 OPRMS 关系**: 独立，纯技术分析系统
> **适用前提**: S 级资产（OPRMS DNA=S），抄底时有基本面信仰兜底

---

## 核心理念

### 系统本质

这不是一个预测方向的系统，而是一个**资本部署时钟**：给定当前技术形态，输出目标仓位百分比。

四个指标覆盖三个维度：
- **PMARP** — 价格拉伸程度（价格维度的波动率）
- **BBWP** — 波动率的百分位（波动率的波动率）
- **RVOL** — 成交量偏离度（参与度维度的波动率）
- **EMA144 斜率** — 趋势速度（方向维度的动量）

方向只占决策的 25%，波动率状态占 75%。

### 三大设计哲学

1. **双重驱动，择优录用**：右侧趋势引擎（最高满仓）+ 左侧反转引擎（最高 40%），取两者较高值，风控 K 仅作用于右侧
2. **不对称确认**：左侧（逆势）依赖"异常"（RVOL 爆量 + BBWP 极值），右侧（顺势）依赖"常态"（EMA 斜率），避免踏空慢牛
3. **交接棒强制清零**：左侧逆势单有保质期，PMARP 回升至中轴后强制归零，严禁底仓抗单死扛

### 核心张力（已识别，不可消除）

> 风控每一次正确地救你于顶部，和每一次错误地把你从趋势中甩出去，用的是同一段代码。

对于 S 级资产：踏空的代价 > 被套的代价。因此风控设计为三档可选，由人工周频切换，而非追求一个"完美参数"。

### 关键术语

| 术语 | 定义 |
|------|------|
| **拐头（向上）** | 当前 K 线指标值 > 上一根 K 线指标值（单根 cross up） |
| **拐头（向下）** | 当前 K 线指标值 < 上一根 K 线指标值（单根 cross down） |
| **状态锁定 (Latch)** | 左侧触发后锁定仓位，直到交接棒或硬止损触发 |
| **交接棒** | 4H PMARP 上穿 50%，左侧仓位强制归零 |

## 模块一：右侧趋势引擎 (Right-Side Engine)

**职能**：顺势重仓捕捉主升浪。

### 信号：EMA144 斜率

斜率 = (当前 EMA144 - 上一根 EMA144) / 上一根 EMA144 × 100%

### 仓位映射：分段线性（连续函数）

**日线战略层 (strategic_mult)**：

```
斜率 < -0.03%    →  0.0（熊市熄火）
-0.03% ~ +0.03%  →  线性插值 0.0 ~ 0.5（震荡过渡）
+0.03% ~ +0.10%  →  线性插值 0.5 ~ 1.0（趋势建立）
> +0.10%          →  1.0（封顶）
```

**4H 战术层 (tactical_mult)**：

```
斜率 < -0.03%    →  0.0
-0.03% ~ +0.03%  →  线性插值 0.0 ~ 1.0
> +0.03%          →  1.0
```

**右侧目标仓位 = strategic_mult × tactical_mult × 100%**

> 注：阈值（0.03%、0.10%）为初始值，BTC 波动率是美股 3-5 倍，需要回测校准。结构确定，参数可调。

## 模块二：左侧反转引擎 (Left-Side Engine)

**职能**：极度恐慌时精准抄底，上限 40% 仓位。
**运作机制**：状态锁定 (Latch)。触发后锁定计算出的仓位，直到交接棒或硬止损。

### ⚠️ 待验证假设

> 本引擎以 `4H PMARP < 2% 且向上拐头` 为必要触发条件。但现有因子研究（`docs/research/2026-03-17-pmarp-crossover-factor-study.md`）显示 **crypto 的 PMARP cross_up 2% 为负向信号**（7d: -0.89%, 30d: -1.29%），与美股相反。
>
> 本系统的差异点在于：(1) 使用 4H 而非日线 (2) 叠加 BBWP + RVOL 双重过滤。这些过滤器能否翻转单因子的负向结论，是 **v1.1 必须验证的核心假设**，不可作为已验证规律直接上线。
>
> **验证方案**：用 Binance 4H 历史数据回测，对比"裸 PMARP cross_up 2%"vs"PMARP + BBWP + RVOL 三重过滤"的 forward return 分布。

### 第一步：点火开关（基础动力分 $x_{base}$）

取值范围：`[0, 2.0]`。**当 $x_{base} = 0$ 时，左侧引擎休眠，跳过后续所有步骤，输出 0%。**

| 条件 | 分数 | 说明 |
|------|------|------|
| 4H PMARP < 2% 且向上拐头 | **1.0** | 必要条件，发令枪 |
| 上条 + 日线 PMARP < 5% | **+1.0** | 战略级超跌共振，满分 2.0 |
| 未触发 | **0** | 左侧休眠，**直接输出 0%** |

### 第二步：质量评估（两大乘数）

仅当 $x_{base} > 0$ 时执行。在发令枪响的同一根 K 线，截取 BBWP 和 RVOL 状态。

**BBWP 波动率乘数 ($M_{bbwp}$)**，取值 `[1.0, 2.5]`：

| 状态 | 乘数 | 说明 |
|------|------|------|
| 日线 BBWP > 98% 且 4H BBWP > 98% 均向下拐头 | **2.5** | 史诗恐慌底 |
| 仅 4H BBWP > 98% 向下拐头（且 PMARP < 10%） | **2.0** | 战术恐慌底 |
| 4H BBWP < 5%（极度挤压） | **1.5** | 阴跌底 |
| 常态 | **1.0** | 无加成 |

**RVOL 成交量乘数 ($M_{rvol}$)**，取值 `[0.5, 2.0]`：

| 状态 | 乘数 | 说明 |
|------|------|------|
| 日线 RVOL > 2.0 且 4H RVOL > 2.5 | **2.0** | 史诗爆量 |
| 仅 4H RVOL > 2.5 | **1.5** | 战术爆量 |
| 日线 RVOL < 0.8（极度缩量） | **0.5** | 阴跌惩罚 |
| 常态 | **1.0** | 无加成 |

### 第三步：仓位计算

1. 总分：$X_{raw} = x_{base} \times M_{bbwp} \times M_{rvol}$
2. 上限锁定：$X = \min(X_{raw}, 4.0)$（**无下限 clamp**，保留 RVOL 缩量惩罚的完整效果）
3. 安全地板：$X = \max(X, 0.25)$（防止公式产出负值，见下方说明）
4. 边际递减映射：$\text{左侧目标仓位} = 40\% \times (0.6 \times \sqrt{X} - 0.2)$
5. 状态锁定：仓位 latch 生效，记录触发价格

> **v1.1 修复说明**：v1.0 的 `clamp(X_raw, 1.0, 4.0)` 有两个问题：
> (1) 当 `x_base=0`（休眠）时，clamp 会把 0 抬到 1，产出 16% 的幽灵仓位 → **修复：x_base=0 时提前返回 0%，不进入公式**
> (2) 下限 1.0 会把 `M_rvol=0.5` 的缩量惩罚完全抹掉（0.5 被抬到 1.0）→ **修复：移除下限 clamp，改用 0.25 安全地板（仅防负值）**

**修复后映射参考值**：

| 场景 | x_base | M_bbwp | M_rvol | X | 左侧仓位 |
|------|--------|--------|--------|---|----------|
| 休眠 | 0 | — | — | — | **0%**（提前返回） |
| 最低信号 + 缩量惩罚 | 1.0 | 1.0 | 0.5 | 0.5 | **9%** |
| 最低信号 + 常态 | 1.0 | 1.0 | 1.0 | 1.0 | **16%** |
| 战术恐慌 + 爆量 | 1.0 | 2.0 | 1.5 | 3.0 | **33%** |
| 战略共振 + 史诗恐慌 + 史诗爆量 | 2.0 | 2.5 | 2.0 | 4.0 (capped) | **40%** |

### 第四步：生命周期管理

**交接棒离场**：4H PMARP 上穿 50% → 左侧仓位归零，引擎熄火，释放 latch。

**硬止损**：价格跌破左侧触发价的 **-20%** → 强制清零，不等交接棒。

## 模块三：风控模块 (Risk Control)

**职能**：对冲右侧引擎在高位的钝化。
**作用域**：⚠️ **仅作用于右侧引擎**，不影响左侧引擎。

> **v1.1 修复说明**：v1.0 的 `MAX(右侧, 左侧) × K` 会在 K=0 时把左侧一起清零。但左侧的核心场景恰恰是日线 EMA 向下的恐慌下跌——此时 K=0，左侧被完全掐死，与系统哲学直接矛盾。
>
> 修复：K 只乘右侧。左侧有独立的风险管理（40% 上限 + latch + 交接棒 + -20% 硬止损），不需要额外的全局 veto。

### 三档模式（人工周频切换）

| | 骑浪 🏄 | 均衡 ⚖️ | 护城 🏰 |
|---|---------|---------|---------|
| **哲学** | 宁可多吃回撤，不踏空 | 攻守平衡 | 宁可磨损，不吃大顶 |
| **适用** | 牛市中段、确信度高 | 默认档 | 疑似见顶、宏观恶化 |
| **触发条件** | PMARP>98% ↓ **且** BBWP>98% ↓（双确认） | PMARP>98% ↓ **或** 高位BBWP↓ | PMARP>90% ↓ 即触发 |
| **K 值** | 0.7 | 0.4 | 0.0 |
| **恢复条件** | 破新高即恢复 | PMARP<50% 或破新高 | PMARP<30% 或（破新高 + 持续3根K线） |

触发时记录当前 K 线最高价为 `逃顶价`，用于破新高恢复判断。

> **时间框架冻结（v1.1）**：风控模块里的 `高位BBWP↓` 默认指 **4H BBWP > 98% 且当前值低于上一根 4H BBWP**。日线 BBWP 只用于左侧质量加分，不参与右侧风控触发。
>
> **边界语义冻结（当前实现）**：本文所有 `>98% / <2% / <5%` 默认表示**严格不含等号**。例如 `PMARP = 98.0` 不算 `>98%`，`PMARP = 2.0` 不算 `<2%`。如果后续要改成含边界，视为显式规则变更，不是实现细节。

### 日线 EMA 方向修正

风控触发后，K 受日线 EMA 状态调制：
- 日线 EMA 斜率 > 0（牛市过热）：K = 模式对应值（右侧被压制，但保留部分仓位）
- 日线 EMA 斜率 ≤ 0（反弹高潮 / 熊市）：K = 0.0（右侧完全清仓）

> 注意：此时左侧引擎不受影响。如果恐慌底信号同时触发，左侧可以独立建仓。

## 系统合成公式

```
右侧_风控后 = 右侧仓位 × K
最终目标仓位 = MAX(右侧_风控后, 左侧仓位)
```

> **v1.1 vs v1.0**：从 `MAX(右, 左) × K` 改为 `MAX(右 × K, 左)`。K 是右侧引擎的"降温阀"，不是全局开关。

## 日常运行 Checklist（每根 4H K 线收盘时）

```
1. 【数据】获取最新 K 线
   ├─ 读取：最新完整 4H K 线 (OHLCV)
   ├─ 读取：最新完整日线（见"数据时序规范"）
   └─ 计算：4 个指标的 4H + 日线值

2. 【风控】状态更新（仅影响右侧）
   ├─ 检查：是否触发新的风控？（按当前模式的触发条件）
   ├─ 检查：已在风控中？恢复条件满足了吗？
   └─ 输出：K 值

3. 【左侧】恐慌底判定（独立于风控）
   ├─ 如果无 latch：PMARP < 2% 且拐头？→ 打分 → latch 锁定 → 记录触发价
   ├─ 如果有 latch：PMARP > 50%？→ 交接棒清零
   ├─ 如果有 latch：价格 < 触发价 × 0.80？→ 硬止损清零
   └─ 输出：左侧仓位（0% 或 latch 值）

4. 【右侧】趋势评估
   ├─ 计算：日线 EMA144 斜率 → strategic_mult
   ├─ 计算：4H EMA144 斜率 → tactical_mult
   ├─ 右侧原始 = strategic × tactical × 100%
   └─ 输出：右侧_风控后 = 右侧原始 × K

5. 【结算】
   └─ 目标仓位 = MAX(右侧_风控后, 左侧仓位)
```

### 执行语义冻结 (v1.1)

为保证 live / backtest 一致，执行口径固定如下：

| 项目 | 规则 |
|------|------|
| **信号计算时点** | 使用刚收盘的 4H bar 数据计算目标仓位 |
| **成交时点** | 下一根 4H bar 的 `open` 执行调仓 |
| **默认交易成本** | 单边 **10 bps** |
| **调仓死区** | 若新目标仓位与当前目标仓位的差值 **< 5 个百分点**，则忽略本次调仓，不扣成本 |
| **基准路径** | Buy & Hold 用 **close-to-close** 累积，不用 open-to-close |

> 示例：49.8% → 50.2% 属于死区内变化，不执行；49.8% → 55.0% 才执行。

## 指标参数冻结 (v1.1 补充)

| 指标 | 参数 | 值 | 说明 |
|------|------|-----|------|
| **EMA** | period | 144 | 日线 + 4H 共用 |
| **PMARP** | ema_period | 20 | 与现有 `src/indicators/pmarp.py` 一致 |
| | lookback | 150 | 150 根 K 线的滚动百分位 |
| **BBWP** | bb_period | 13 | Bollinger Band 周期 |
| | bb_std | 1.0 | BB 标准差倍数 |
| | lookback | 252 | BBW 的百分位回看期（约 1 年日线 / 42 天 4H） |
| **RVOL** | window | 20 | 滚动均值/标准差窗口 |

> BBWP 定义：`BBW = (Upper - Lower) / Middle`，`BBWP = Percentile(BBW, lookback)`
> RVOL 定义：`RVOL = (Volume - Mean(Volume, window)) / StdDev(Volume, window)`

> ⚠️ 以上参数为初始值，基于 TradingView 社区常用设置。BTC 4H 的最优参数需回测确认，但结构先冻结。

## 数据时序规范 (v1.1 补充)

### 问题定义

4H 轮询时需要"日线指标"，但当天日线可能尚未收盘。如果读未收盘日线，live 和 backtest 的口径不一致。

### 规范

| 场景 | 日线数据来源 | 说明 |
|------|-------------|------|
| **Live（实时轮询）** | 上一根**已收盘**的完整日线 | 不使用当日未收盘数据，保证确定性 |
| **Backtest（回测）** | 同上，只使用 `date < current_4h_bar_date` 的日线 | 严格防前视偏差 |

4H K 线边界（UTC）：00:00 / 04:00 / 08:00 / 12:00 / 16:00 / 20:00
日线边界（Binance）：UTC 00:00

因此：UTC 00:00 的 4H bar 收盘时，恰好有一根新的完整日线可用。其余 5 根 4H bar 使用前一天的日线。

### 状态持久化与审计

系统运行需要跨 K 线保持的状态：

| 状态 | 类型 | 持久化 |
|------|------|--------|
| `risk_mode` | enum: surf / balanced / fortress | 用户手动设置，存配置文件 |
| `risk_active` | bool | 风控是否已触发 |
| `escape_price` | float | 风控触发时记录的最高价 |
| `K` | float | 当前风控系数 |
| `left_latch_active` | bool | 左侧是否处于锁定状态 |
| `left_latch_position` | float | 锁定的仓位百分比 |
| `left_trigger_price` | float | 左侧触发时的价格（硬止损基准） |

存储方式冻结为 **SQLite**，每根 4H K 线更新后写入。

| 表 | 用途 | 说明 |
|----|------|------|
| `engine_state` | 当前状态表 | 每个 system_name 保留最新状态，供实时运行恢复 |
| `engine_history` | 审计表 | 每根 4H bar 追加/幂等记录一次评估结果，供复盘与审计 |

`engine_history` 至少保存：`engine_timestamp`、目标仓位、左右引擎仓位、`K`、触发原因、risk/latch 状态、触发价。

## 验证协议 (v1.1 补充)

在系统上线前必须完成的验证：

### V1：核心假设验证（左侧 PMARP 触发条件）

**假设**：4H PMARP < 2% cross_up + BBWP/RVOL 过滤，在 BTC 上产生正向 forward return。

**方法**：
1. 获取 BTC 4H 历史数据（≥3 年）
2. 回测"裸信号"：仅 4H PMARP < 2% cross_up → 统计 7d/30d forward return
3. 回测"过滤信号"：裸信号 + BBWP/RVOL 条件 → 统计同上
4. 对比 (2) vs (3)，过滤器是否显著改善 hit rate 和 mean forward return

**通过标准**：过滤后信号的 30d mean forward return > 0，且优于裸信号，p < 0.10（探索性研究，放宽阈值）

### V2：右侧引擎参数校准

校准 EMA144 斜率阈值（0.03% / 0.10%），使用 BTC 日线 + 4H 历史数据。

### V3：全系统回测

三档模式全部回测，输出：
- 总收益 vs Buy & Hold
- 最大回撤
- 趋势捕捉率（= 系统在 BTC 上涨月份中的平均仓位）
- 磨损率（= 风控触发后又回补的次数 / 总风控触发次数）
